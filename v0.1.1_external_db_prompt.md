# Gym Tracker v0.1.1 — External Database Support

## Overview

Currently, the app copies a bundled `gym_tracker.db` into its internal sandboxed storage on first launch and never touches it again. The Import feature replaces this internal copy with a user-selected file. The Export feature copies it to a temp directory for sharing.

**Problem:** Once imported, the live database is locked inside the app's private storage. External tools (like an AI coach managing the same data) can't access it without re-importing/exporting every time.

**Solution:** Add an "External Database" mode where the app opens a database at a user-specified filesystem path directly — no copying, no internalization. This makes the database a true shared resource that both the app and external tools can work with simultaneously.

Three modes, user-selectable:

1. **Internal DB (default)** — current behavior, unchanged. Fresh internal DB from bundled asset for new users.
2. **External DB** — user picks a `.db` file via file picker. App persists the path and opens that file directly. No copying. The app reads/writes to the external file in place.
3. **Export internal DB** — users on the internal DB can export it as a `.db` file (file picker "save" dialog) in addition to the existing share-via-email/etc flow.

## Part 2 — DatabaseHelper: External DB Path Support

### 2.1 Add SharedPreferences dependency

Add to `pubspec.yaml` under dependencies:
```yaml
  shared_preferences: ^2.3.0
```

Run `flutter pub get`.

### 2.2 Modify `DatabaseHelper` in `lib/database/database_helper.dart`

Add:
- A constant key for persisting the external DB path: `_prefsKeyExternalDbPath = 'external_db_path'`
- A static field `_externalDbPath` to cache the current external path (or null for internal mode)
- A static field `_dbPath` to cache the resolved database path (so we don't recompute it on every `database` getter call)

**New public methods:**

```dart
/// Whether the database is currently using an external file.
bool get isExternal => _externalDbPath != null && _externalDbPath!.isNotEmpty;

/// Returns the path to the external database file, or null if using internal.
String? get externalDbPath => _externalDbPath;

/// Resolves the database path: external path if set, otherwise internal.
Future<String> _resolvePath() async { ... }

/// Sets the database to use an external file at [path].
/// Closes any open connection, persists the preference, and reopens.
Future<String> setExternalDatabase(String path) async { ... }

/// Reverts to the internal database.
/// Closes any open connection, clears the persisted external path, and reopens.
/// If the internal DB doesn't exist yet, it will be created from assets on next open.
Future<void> useInternalDatabase() async { ... }

/// Loads a persisted external DB path from SharedPreferences on startup.
/// Called once before the first database access.
Future<void> loadPersistedPath() async { ... }
```

**Changes to `_initDatabase()`:**

Currently:
```dart
final path = join(databasesPath, _databaseName);
```

Replace with logic that:
1. Calls `loadPersistedPath()` if not yet loaded
2. If an external path is persisted, uses that path directly (skip asset copy, skip default getDatabasesPath())
3. If no external path, falls back to the current internal behavior (getDatabasesPath + asset copy if not exists)
4. Stores the resolved path in `_dbPath` for later reference

**Validation in `setExternalDatabase()`:**
- Verify the file at `path` exists
- Open it read-only and validate it's a v3 Gym Tracker DB (reuse existing `_validateV3Database` logic)
- If valid, close current connection, persist the path to SharedPreferences, set `_externalDbPath`, and reopen
- If invalid, return a descriptive error — do NOT change the current DB state

**Error handling:**
- If the external file is deleted or moved while the app is closed, `_initDatabase()` should detect the missing file and show an error state rather than silently creating a new internal DB. But for v0.1.1, a simple check: if external path is set but file doesn't exist, log a warning and fall back to internal. We can add a proper error dialog in a later version.

## Part 3 — GymRepository: Expose External DB State

In `lib/repositories/gym_repository.dart`, add:

```dart
/// Whether the database is currently external (not internal).
bool get isExternal => _databaseHelper.isExternal;

/// Path to the external database file, or null.
String? get externalDbPath => _databaseHelper.externalDbPath;

/// Sets the database to use an external file.
Future<String> setExternalDatabase(String path) async {
  return await _databaseHelper.setExternalDatabase(path);
}

/// Reverts to the internal database.
Future<void> useInternalDatabase() async {
  await _databaseHelper.useInternalDatabase();
}
```

## Part 5 — Android-Specific Considerations

### 5.1 External Storage Path

On Android, the external DB should be stored in a location accessible to both the app and file managers. The cleanest approach for Android 10+ is:

- Use `getExternalStorageDirectory()` from `path_provider` if the file is within app-specific external storage
- For truly shared locations (Download, Documents), use SAF (Storage Access Framework) via `file_picker`

For v0.1.1, keep it simple: the user picks a file using the file picker. If they can navigate to it and select it, the app can open it (file picker handles the SAF permissions on Android). The path returned by `file_picker` on Android may be a content:// URI — if so, copy the file to a temporary location first, then use that copy's path. Add a note in the code that this is a known limitation and will be improved in a future version.

Actually, `sqflite` on Android works with file paths, not content:// URIs. So for Android, when the user picks a file:
1. Copy the picked file to a known accessible path (e.g., `getExternalStorageDirectory()/gym_tracker_external.db`)
2. Use that path as the external DB path
3. Note: this means the "external DB" on Android is actually a copy in app-specific storage — not truly shared. But it's still accessible via the export feature.

**Better Android approach (recommended):** Instead of SAF, document that on Android the external DB should be placed in `Documents/` or `Download/` which are accessible via `getExternalStorageDirectory()` or `getApplicationDocumentsDirectory()`. The file picker can browse these locations on Android 10+ with `requestLegacyExternalStorage`.

For now, implement the simplest working version: file picker → copy to app-accessible path → open. This works on all platforms.

### 5.2 No AndroidManifest Changes Needed for v0.1.1

Since we'll copy the picked file to app storage, we don't need the storage permissions for the external DB feature itself. The existing `READ_EXTERNAL_STORAGE` for import is sufficient. Add `WRITE_EXTERNAL_STORAGE` only if implementing the save-file export flow (Part 4.4).

## Design Decisions & Rationale

**Why not just always use an external path?** Because it breaks the zero-config first-launch experience. A new user downloads the app, opens it, and it works — they don't need to know about file paths.

**Why persist the external path in SharedPreferences?** So the choice survives app restarts. Otherwise the user would need to re-select the external DB every time they open the app.

**Why validate on setExternalDatabase?** Because opening a corrupted or non-v3 file could crash the app or corrupt data. The existing `_validateV3Database` method already does this check — we reuse it.

**Why copy on Android instead of opening the picked file directly?** `sqflite` requires a filesystem path, but Android's SAF file picker returns a content:// URI. Copying to app storage is the simplest workaround. A future version could use SAF properly with a document provider.

**What happens if the external file is deleted while the app is running?** The app will error on the next database operation. For v0.1.1, this is acceptable — the user has to be deliberate about deleting their database file. A future version could watch the file and offer to switch back to internal mode.
